## 0. 작업자(개발자/오너)에 대한 안내 — 반드시 지킬 것

### 0-1. 기본 스타일

- 오너는 코드를 깊게 다루지 않는 **바이브 코딩** 스타일이다.
- 코드를 짤 때 **무엇을, 왜 그렇게 짰는지를 쉬운 말로 같이 설명**한다. (단순 코드 덤프 금지)
- 큰 변경 전에는 **무엇을 바꿀지 먼저 말하고 동의를 구한 뒤** 실행한다.
- 파일을 여러 개 한 번에 쏟아내기보다, **한 덩어리씩 만들고 확인받으며** 진행한다.

### 0-2. 핵심 개념 교육 (바이브만으론 배우지 못하는 것들)

바이브 코딩만으로는 배움이 없다. 다음 개념들은 **코드 구현 과정에서 자연스럽게 설명하되, "이 정도는 알아야 한다"는 수준을 지킨다.**

**필수 개념 (꼭 이해해야 함)**

1. **어댑터 패턴 (Adapter Pattern)**
   - 왜 중요: PICKIT의 핵심 아키텍처. 6개 채널이 각각 다른 API를 써도 PICKIT이 단일 인터페이스로 제어할 수 있는 이유.
   - 설명 시점: `src/connectors/` 폴더 첫 생성 시
   - 쉬운 설명: "각 채널(쿠팡, 네이버 등)은 다 다르지만, PICKIT은 그들을 같은 언어로 제어한다. 그 '번역기' 역할이 어댑터야."

2. **API 구조 (REST API, 요청-응답)**
   - 왜 중요: 백엔드와 프론트를 잇는 통신 방식. PICKIT의 명령 흐름 전체를 이해하려면 필수.
   - 설명 시점: 첫 API 라우트 만들 때 (예: `GET /orders`)
   - 쉬운 설명: "사용자가 '주문 보여줘' → 프론트가 백엔드에 요청(Request) 보냄 → 백엔드가 DB에서 꺼내서 응답(Response) 줌"

3. **비동기 처리 (async/await, Promise)**
   - 왜 중요: 외부 API 호출(쿠팡, 네이버 등)은 시간이 걸린다. 그 동안 다른 작업을 막지 않으려면 필수.
   - 설명 시점: 첫 채널 API 호출할 때
   - 쉬운 설명: "커피를 주문하고(async) 받을 때까지 기다리는 동안(await) 다른 손님 응대할 수 있게. 동기면 한 명씩 처리하고 기다려야 함."

4. **상태 관리 (React State, Props)**
   - 왜 중요: 채팅창에서 "주문 조회 완료", "방화벽 대기 중" 같은 UI 상태 변화를 제어.
   - 설명 시점: ChatWidget.jsx에서 첫 상태 변수 만들 때
   - 쉬운 설명: "state는 컴포넌트의 '기억'. '지금 로딩 중'을 state에 저장하면 UI가 그에 맞게 자동 변함."

5. **방화벽 로직 (조건문, 검증)**
   - 왜 중요: 위험한 수정을 사전에 막는 PICKIT의 핵심 안전장치. 조건문과 검증의 실제 사용.
   - 설명 시점: `checkSafetyGuardrail()` 구현할 때
   - 쉬운 설명: "if(가격이 20% 이상 내렸으면) 멈추고 승인 받기. 사람이 말한 명령 그대로 실행하면 위험하니까."

6. **환경 변수와 보안 (API Key, Token 저장)**
   - 왜 중요: API 키를 코드에 직접 쓰면 안 돼. GitHub 올려질 수 있으니까. 이게 보안의 기본.
   - 설명 시점: 첫 API 키 사용할 때
   - 쉬운 설명: "비밀번호를 코드에 써놓고 세상에 공개하는 것과 같음. `.env` 파일에 숨겨놔야 함."

**선택 개념 (알면 좋지만, 나중에 배워도 됨)**
- 데이터베이스 쿼리 (SELECT, WHERE, 인덱스)
- 스키마 설계 (테이블 구조)
- 에러 처리 (try-catch, 타임아웃)
- 배포 (환경 변수, 프로덕션 vs 개발)

### 0-3. 설명 방식

- 코드 라인 번호로 지정해서 "이 부분에서 async/await를 썼는데..." 이렇게 정확히.
- 한 줄짜리 용어 설명 아니라, **작은 예시 + 왜 필요한지 2~3줄**.
- 구현이 끝난 후 "이제 너는 [개념]을 알게 됐다"고 정리.
- 오너가 "이 부분 이해 못 했어" 하면 **도표나 흐름도** 그려서라도 설명.

## 1. 제품 한 줄 정의

PICKIT = 의류 멀티채널 셀러를 위한 **자연어 기반 멀티채널 운영 서비스**.
"엑셀 관리는 한계, ERP는 아직 부담"인 소규모 셀러가 타깃.
채팅창에 말 한마디 → 6개 채널의 주문 조회·수정·송장·재고를 한 번에 처리.

## 2. 지원 채널 (6개) — 기능 차등 주의

| 그룹 | 채널 | MVP 권한 |
| --- | --- | --- |
| 핵심 본진 | 쿠팡 / 네이버 스마트스토어 / 카페24 | **조회·수정·송장·동기화 전부 지원** |

## 3. 기술 스택

> 아래는 PRD에서 드러난 방향. **확정 안 된 항목은 임의로 정하지 말고 오너에게 먼저 물어볼 것.**

## 4. 절대 지켜야 할 아키텍처 5규칙 (위반 금지)

### 규칙 ① AI와 쇼핑몰 API의 완전 격리 (어댑터 패턴)
- AI에게는 **단 하나의 함수**만 Tool로 준다:
  `modify_product_data(platform, productId, action, value)`
- 백엔드가 `platform` 값으로 분기 → `src/connectors/coupang.js` 등 실제 커넥터 호출
- **신규 채널 추가 = `src/connectors/`에 파일 하나 추가**. AI·코어 코드는 건드리지 않는다.
- (쉬운 설명: AI는 "주문해"라고만 말하고, 실제 요리는 채널별 주방이 따로 한다. 주방 하나 늘려도 손님 응대 방식은 안 바뀐다.)

### 규칙 ② 방화벽 파이프라인 (안전장치, 절대 우회 금지)
- AI가 만든 수정 값은 **실제 API 전송 직전 반드시 `checkSafetyGuardrail()` 통과**.
- 4종 방화벽:
  1. **금액**: 기존가 대비 ±20% 초과 또는 0원 이하 → 보류
  2. **범위**: 대상 5개 이상 또는 '전체/모두/싹 다/일괄' 키워드 → 보류
  3. **맥락(확신도)**: Confidence < 85% 또는 정보 부족 → 되물음
  4. **플랫폼 불일치**: 미지원 기능 요청 → 가능 채널만 처리 + 불가 채널은 링크
- 트리거 시: `isDangerous: true` 반환 → 실행 Hold → 채팅 내 `[최종 승인] / [취소]` 버튼 렌더 → **승인 전 API 호출 금지**

### 규칙 ③ 연동 예외 처리 & 기본값 필터
- 연동 에러 코드 수신 → 해당 채널만 Hold + 주황색 경고 + [연동 설정] 링크
- 기간 키워드 없는 조회는 **자동으로 당일(오늘 00:00~현재)만** 호출 (서버 보호)

### 규칙 ④ 프론트 스타일 격리 & 연산 분산
- `ChatWidget.jsx`는 CSS Modules 또는 Styled-components로 스타일 격리 + `position: fixed; z-index: 9999;`
- 엑셀은 백엔드가 깨끗한 JSON만 주고, **파일 생성은 프론트(셀러 PC)에서** → 서버 비용 절감

### 규칙 ⑤ 데이터 스키마 분리
- `sales_log`: 상품별 집계 판매량 시계열 (개인정보 X, 발주 예측용) → **MVP 배포 즉시 적재 시작 (최우선)**
- `customer_purchase`: 회원별 구매 이력 (Post-MVP CRM용) → 지금은 만들지 않음
- 원칙: **예측은 기능이 아니라 쌓인 데이터의 결과물.** sales_log 적재가 늦으면 예측도 영구히 늦어진다.

## 5. 보안 / 비기능 규칙

- API Key·Token은 **서버 측 암호화 저장**. 절대 프론트·git·로그에 노출 금지.
- 주문 데이터는 장기 캐시하지 않고 실시간 조회 원칙.
- 외부 API 실패 시 최대 3회 재시도 후 에러 처리.
- 한 채널 장애가 전체 서비스 장애로 번지지 않게 **채널별 상태 격리**.
- 일반 조회 응답 목표 3초 이내.

## 6. 구현 우선순위 (MVP)

- **P0 (먼저)**: FR-001 온보딩 / FR-002 주문 조회 / FR-003 품절·가격 수정 / FR-004 송장 전송 / FR-007 방화벽
- **P1 (다음)**: FR-005 엑셀 추출 / FR-006 발주 예측 / FR-008 연동 끊김 안내 / FR-009 재고 동기화

> 추천 구현 순서: ① 방화벽 + 어댑터 골격(②④①규칙) → ② 주문 조회(READ) → ③ 수정(WRITE, 방화벽 연결) → ④ 송장 → ⑤ 나머지 P1
