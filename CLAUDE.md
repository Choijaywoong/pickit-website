## 0. 작업자(개발자/오너)에 대한 안내 — 반드시 지킬 것

### 0-1. 기본 스타일

- 오너는 코드를 깊게 다루지 않는 **바이브 코딩** 스타일이다.
- 코드를 짤 때 **무엇을, 왜 그렇게 짰는지를 쉬운 말로 같이 설명**한다. (단순 코드 덤프 금지)
- 큰 변경 전에는 **무엇을 바꿀지 먼저 말하고 동의를 구한 뒤** 실행한다.
- 파일을 여러 개 한 번에 쏟아내기보다, **한 덩어리씩 만들고 확인받으며** 진행한다.
- 가정은 숨기지 말고 명시한다. 해석이 여러 갈래면 임의로 하나 고르지 말고 제시한 뒤 묻는다. (→ 9.1 Think Before Coding과 연결)

### 0-2. 핵심 개념 교육 (바이브만으론 배우지 못하는 것들)

바이브 코딩만으로는 배움이 없다. 다음 개념들은 **코드 구현 과정에서 자연스럽게 설명하되, "이 정도는 알아야 한다"는 수준을 지킨다.**

**필수 개념 (꼭 이해해야 함)**

1. **어댑터 패턴 (Adapter Pattern)**
   - 왜 중요: PICKIT의 핵심 아키텍처. 6개 채널이 각각 다른 API를 써도 PICKIT이 단일 인터페이스로 제어할 수 있는 이유.
   - 설명 시점: `src/connectors/` 폴더 첫 생성 시
   - 쉬운 설명: "각 채널(쿠팡, 네이버 등)은 다 다르지만, PICKIT은 그들을 같은 언어로 제어한다. 그 '번역기' 역할이 어댑터야."

2. **API 구조 (REST API, 요청-응답)**
   - 왜 중요: 백엔드와 프론트를 잇는 통신 방식. PICKIT의 명령 흐름 전체를 이해하려면 필수.
   - 설명 시점: 첫 API 라우트 만들 때 (예: `GET /orders`)
   - 쉬운 설명: "사용자가 '주문 보여줘' → 프론트가 백엔드에 요청(Request) 보냄 → 백엔드가 DB에서 꺼내서 응답(Response) 줌"

3. **비동기 처리 (async/await, Promise)**
   - 왜 중요: 외부 API 호출(쿠팡, 네이버 등)은 시간이 걸린다. 그 동안 다른 작업을 막지 않으려면 필수.
   - 설명 시점: 첫 채널 API 호출할 때
   - 쉬운 설명: "커피를 주문하고(async) 받을 때까지 기다리는 동안(await) 다른 손님 응대할 수 있게. 동기면 한 명씩 처리하고 기다려야 함."

4. **상태 관리 (React State, Props)**
   - 왜 중요: 채팅창에서 "주문 조회 완료", "방화벽 대기 중" 같은 UI 상태 변화를 제어.
   - 설명 시점: ChatWidget.jsx에서 첫 상태 변수 만들 때
   - 쉬운 설명: "state는 컴포넌트의 '기억'. '지금 로딩 중'을 state에 저장하면 UI가 그에 맞게 자동 변함."

5. **방화벽 로직 (조건문, 검증)**
   - 왜 중요: 위험한 수정을 사전에 막는 PICKIT의 핵심 안전장치. 조건문과 검증의 실제 사용.
   - 설명 시점: `checkSafetyGuardrail()` 구현할 때
   - 쉬운 설명: "if(가격이 20% 이상 내렸으면) 멈추고 승인 받기. 사람이 말한 명령 그대로 실행하면 위험하니까."

6. **환경 변수와 보안 (API Key, Token 저장)**
   - 왜 중요: API 키를 코드에 직접 쓰면 안 돼. GitHub 올려질 수 있으니까. 이게 보안의 기본.
   - 설명 시점: 첫 API 키 사용할 때
   - 쉬운 설명: "비밀번호를 코드에 써놓고 세상에 공개하는 것과 같음. `.env` 파일에 숨겨놔야 함."

7. **LLM은 '생성기'지 '계산기'가 아니다 (예측·숫자 처리의 경계)**
   - 왜 중요: PICKIT의 핵심 무기인 발주 예측을 LLM에게 맡기면 안 되는 이유. 규칙 ⑥의 근거.
   - 설명 시점: 발주 예측(FR-006) 또는 LLM 응답 생성 코드를 만질 때
   - 쉬운 설명: "LLM은 '그럴듯한 다음 말'을 만드는 기계지, '숫자를 계산하는' 기계가 아니야. '이 판매 데이터로 언제 품절될지 예측해줘' 하면 그럴듯한 숫자를 뱉지만 그건 계산이 아니라 추측이고, 같은 데이터로 또 물으면 다른 답이 나올 수도 있어(=환각). 셀러는 이 숫자 믿고 진짜 돈 써서 발주하니까, 예측은 반드시 통계 계산으로 하고 LLM은 그 결과를 '말로 풀어 설명'하는 데만 쓴다."

8. **카테고리는 '입력'이 아니라 '데이터의 결과'다 (단일 예측 모델)**
   - 왜 중요: 발주 예측을 카테고리별로 나누지 않고 단일 엔진으로 짜는 이유. 규칙 ⑤의 근거.
   - 설명 시점: 발주 예측 엔진(`src/forecast/engine.js`) 만질 때
   - 쉬운 설명: "옷이 환절기에 잘 팔리는지, 화장품이 45일마다 재구매되는지는 이미 그 상품의 판매 기록(sales_log)에 다 들어있어. 그러니 사람이 '넌 의류니까 1.3배' 같은 규칙을 손으로 박을 필요가 없어. 데이터가 직접 말하게 두면 카테고리 라벨을 몰라도 예측이 돼. 카테고리는 예측에 넣는 재료가 아니라, 데이터를 보면 저절로 드러나는 결과야."

**선택 개념 (알면 좋지만, 나중에 배워도 됨)**
- 데이터베이스 쿼리 (SELECT, WHERE, 인덱스)
- 스키마 설계 (테이블 구조)
- 에러 처리 (try-catch, 타임아웃)
- 배포 (환경 변수, 프로덕션 vs 개발)

### 0-3. 설명 방식

- 코드 라인 번호로 지정해서 "이 부분에서 async/await를 썼는데..." 이렇게 정확히.
- 한 줄짜리 용어 설명 아니라, **작은 예시 + 왜 필요한지 2~3줄**.
- 구현이 끝난 후 "이제 너는 [개념]을 알게 됐다"고 정리.
- 오너가 "이 부분 이해 못 했어" 하면 **도표나 흐름도** 그려서라도 설명.

## 1. 제품 한 줄 정의

PICKIT = **자연어 기반 멀티채널 운영 서비스**.
채팅창에 말 한마디 → 6개 채널의 주문 조회·수정·송장·재고를 한 번에 처리.

**타깃 / 진입 전략 (코드에 영향 주는 부분만):**
- 1차 진입 시장(GTM)은 **의류 멀티채널 셀러**다. 메시징·온보딩 예시는 의류 기준.
- 다만 **제품 아키텍처는 카테고리 중립(vertical-neutral)으로 유지한다.** 즉 "의류 전용"으로 기능을 하드코딩하지 않는다. 의류는 첫 고객일 뿐, 뷰티 등 다른 카테고리로 확장 가능한 구조로 짠다.
- 핵심 사용자 규모는 "엑셀 졸업 ~ 중소형 브랜드"다. 단일 채널 셀러는 대상이 아니다(통합 가치 없음).

> **카테고리 중립 원칙**: 카테고리는 예측·기능의 **입력이 아니라 데이터의 결과**다. 카테고리별로 기능을 하드코딩하거나 `if (category === ...)`로 분기하지 않는다. 카테고리에 따른 차이(계절성·재구매주기 등)는 **각 상품의 `sales_log`에서 자동 추출**한다. (→ 규칙 ⑤)

## 3. 기술 스택

> 아래는 PRD에서 드러난 방향. **확정 안 된 항목은 임의로 정하지 말고 오너에게 먼저 물어볼 것.**

- 프론트엔드: React (채팅 위젯 `ChatWidget.jsx` 중심)
- 백엔드: Node.js (커넥터 + 방화벽 인터셉터 + LLM Tool 핸들러 + **예측 엔진**)
- LLM: Tool Calling(Function Calling) 방식으로 AI에 단일 마스터 함수만 노출. **예측 계산은 시키지 않는다(규칙 ⑥).**
- 예측: `sales_log` 기반 **단일 통계 엔진**(이동평균 + 데이터 기반 계절 보정). 카테고리 무관. LLM 아님.
- 엑셀 생성: 프론트엔드 `xlsx` 라이브러리 (서버에 파일 저장 안 함)
- DB: `sales_log`, `customer_purchase` 테이블 (DB 종류 미확정 — 물어볼 것)

## 4. 절대 지켜야 할 아키텍처 6규칙 (위반 금지)

### 규칙 ① AI와 쇼핑몰 API의 완전 격리 (어댑터 패턴)
- AI에게는 **단 하나의 함수**만 Tool로 준다:
  `modify_product_data(platform, productId, action, value)`
- 백엔드가 `platform` 값으로 분기 → `src/connectors/coupang.js` 등 실제 커넥터 호출
- **신규 채널 추가 = `src/connectors/`에 파일 하나 추가**. AI·코어 코드는 건드리지 않는다.
- (쉬운 설명: AI는 "주문해"라고만 말하고, 실제 요리는 채널별 주방이 따로 한다. 주방 하나 늘려도 손님 응대 방식은 안 바뀐다.)

### 규칙 ② 방화벽 파이프라인 (안전장치, 절대 우회 금지)
- AI가 만든 수정 값은 **실제 API 전송 직전 반드시 `checkSafetyGuardrail()` 통과**.
- 4종 방화벽:
  1. **금액**: 기존가 대비 ±20% 초과 또는 0원 이하 → 보류
  2. **범위**: 대상 5개 이상 또는 '전체/모두/싹 다/일괄' 키워드 → 보류
  3. **맥락(확신도)**: Confidence < 85% 또는 정보 부족 → 되물음
  4. **플랫폼 불일치**: 미지원 기능 요청 → 가능 채널만 처리 + 불가 채널은 링크
- 트리거 시: `isDangerous: true` 반환 → 실행 Hold → 채팅 내 `[최종 승인] / [취소]` 버튼 렌더 → **승인 전 API 호출 금지**

### 규칙 ③ 연동 예외 처리 & 기본값 필터
- 연동 에러 코드 수신 → 해당 채널만 Hold + 주황색 경고 + [연동 설정] 링크
- 기간 키워드 없는 조회는 **자동으로 당일(오늘 00:00~현재)만** 호출 (서버 보호)

### 규칙 ④ 프론트 스타일 격리 & 연산 분산
- `ChatWidget.jsx`는 CSS Modules 또는 Styled-components로 스타일 격리 + `position: fixed; z-index: 9999;`
- 엑셀은 백엔드가 깨끗한 JSON만 주고, **파일 생성은 프론트(셀러 PC)에서** → 서버 비용 절감

### 규칙 ⑤ 데이터 스키마 분리 & 발주 예측 아키텍처
- **데이터 스키마 분리**
  - `sales_log`: 상품별 집계 판매량 시계열 (개인정보 X, 발주 예측용) → **MVP 배포 즉시 적재 시작 (최우선)**
  - `customer_purchase`: 회원별 구매 이력 (Post-MVP CRM용) → 지금은 만들지 않음
  - 원칙: **예측은 기능이 아니라 쌓인 데이터의 결과물.** sales_log 적재가 늦으면 예측도 영구히 늦어진다.
- **발주 예측은 카테고리 무관 '단일 예측 엔진'으로 짠다** (가중치를 카테고리별로 나누지 않는다)
  - **하나의 엔진**: "판매 속도(최근 N일 판매량 ÷ 일수) → 재고 ÷ 판매속도 = 소진 예상일"이라는 본체는 옷이든 화장품이든 동일하다. 엔진은 단 하나(`src/forecast/engine.js`)만 만든다.
  - **계절성·추세·재구매주기 같은 보정값은 사람이 카테고리별로 박는 상수가 아니라, 각 상품의 `sales_log`에서 자동 추출한다.** (예: "이 상품이 작년 같은 시기에 얼마 팔렸나"를 데이터가 직접 말하게 둔다.)
  - **금지**: 코어 예측 로직 안의 `if (category === '의류')` 분기, 카테고리별 가중치 상수 테이블, 사람이 손으로 정한 카테고리 배수. (PICKIT 철학 "예측은 데이터의 결과물"과 정면충돌)
  - **콜드 스타트(데이터 없는 신상품)**: 단일 모델의 유일한 약점. 과거 판매가 없으면 시계열을 못 뽑는다. → "예측 보류(데이터 누적 중 안내)" 또는 "유사 상품 그룹 평균으로 임시 추정" 중 택1. **이때도 카테고리 라벨을 하드코딩하지 말고, 실제 판매 패턴이 비슷한 상품 그룹으로 묶는다.**
  - **사용기한·선입선출 같은 카테고리 제약은 '예측'이 아니라 '발주 권장량 계산 시의 별도 안전 규칙'이다.** 예측 엔진 밖에서 따로 처리해 엔진을 깨끗하게 유지한다.
  - (쉬운 설명: 예측 '엔진'은 한 대뿐이고, 계절·재구매 같은 특성은 각 상품 데이터에서 그때그때 뽑아 쓴다. "넌 의류니까 이만큼"이라고 사람이 미리 정하지 않는다.)

### 규칙 ⑥ 예측 계산(통계) ↔ 자연어 설명(LLM)의 분리 (위반 금지)
- **발주 예측의 숫자는 절대 LLM에게 계산시키지 않는다.** LLM은 생성기지 계산기가 아니다(→ 0-2의 7번).
- 역할 분담을 코드 레벨에서 강제한다:
  - **숫자(소진 예상일, 권장 발주 수량 등) = 통계 엔진(`src/forecast/engine.js`)이 계산.** 같은 입력 → 같은 출력, 검증 가능.
  - **자연어 설명 = LLM이 담당.** 단, LLM에는 **이미 계산이 끝난 숫자를 넘겨주고 "말로 풀어 설명"만** 시킨다. 예측 자체를 LLM에 위임 금지.
- 예: 통계 엔진이 `{소진일: '2026-10-03', 권장수량: 80}` 산출 → LLM이 "○○ L사이즈, 추석 수요까지 감안하면 10월 초 소진 예상이라 지금 80개 발주를 권해요"로 변환.
- 이유: 셀러는 이 숫자를 믿고 실제 돈을 써서 발주한다. 틀린 예측 = 악성재고 또는 품절 손실 = 셀러 손해 = PICKIT 신뢰 즉사. 방화벽(규칙 ②)으로 안전을 강조하면서 예측을 LLM 추측에 맡기는 것은 자기모순이다.
- 테스트 성공 기준: 동일한 `sales_log` 입력에 대해 예측 엔진이 **항상 동일한 숫자**를 반환한다(결정론적). 무작위성·LLM 호출이 숫자 산출 경로에 끼어들면 실패.

# 9. 공통 코딩 행동 가이드라인

LLM 코딩 실수를 줄이기 위한 공통 가이드라인. 위 PICKIT 고유 규칙(0~8)과 함께 적용한다.

**Tradeoff:** 이 가이드라인은 속도보다 신중함 쪽으로 치우쳐 있다. 사소한 작업은 판단껏 한다.

> 참고: **9.1(Think Before Coding)의 일부는 0-1에 이미 반영**돼 있다. 0-1을 우선 적용하고, 9.1은 그 보완으로 본다.

## 9.1. Think Before Coding

**가정하지 마라. 혼란을 숨기지 마라. 트레이드오프를 드러내라.**

구현 전에:
- 가정을 명시한다. 불확실하면 묻는다.
- 해석이 여러 갈래면 제시한다 — 말없이 하나 고르지 않는다.
- 더 단순한 방법이 있으면 말한다. 필요하면 반대 의견도 낸다.
- 불명확하면 멈춘다. 뭐가 헷갈리는지 짚고 묻는다.

## 9.2. Simplicity First

**문제를 푸는 최소한의 코드만. 미리 짐작한 건 넣지 마라.**

- 요청 안 된 기능은 만들지 않는다.
- 한 번만 쓰는 코드에 추상화를 만들지 않는다.
- 요청 안 된 '유연성'이나 '설정 가능성'은 넣지 않는다.
- 일어날 수 없는 시나리오에 대한 에러 처리는 넣지 않는다.
- 200줄을 짰는데 50줄로 줄일 수 있으면 다시 짠다.

스스로에게 묻기: "시니어 엔지니어가 이걸 보고 과하다고 할까?" 그렇다면 단순화한다.

> 발주 예측도 이 원칙을 따른다. 카테고리별 가중치 플러그인·분기 같은 "미리 짐작한 유연성"을 만들지 않는다. 엔진은 하나, 카테고리 차이는 데이터에서 뽑는다(규칙 ⑤). 이게 가장 단순하면서 PICKIT 철학에도 맞는 구조다.

## 9.3. Surgical Changes

**꼭 건드려야 할 것만 건드린다. 네가 만든 쓰레기만 치운다.**

기존 코드를 수정할 때:
- 인접한 코드·주석·포맷을 '개선'하지 않는다.
- 망가지지 않은 걸 리팩터링하지 않는다.
- 다르게 하고 싶더라도 기존 스타일을 따른다.
- 관련 없는 죽은 코드를 발견하면 — 지우지 말고 언급만 한다.

네 변경이 고아 코드를 만들면:
- **네 변경이** 안 쓰게 만든 import·변수·함수는 제거한다.
- 원래 있던 죽은 코드는 요청받지 않는 한 제거하지 않는다.

테스트: 바뀐 모든 줄이 오너의 요청과 직접 연결돼야 한다.

## 9.4. Goal-Driven Execution

**성공 기준을 정의한다. 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 바꾼다:
- "검증 추가" → "잘못된 입력에 대한 테스트를 쓰고, 통과시킨다"
- "버그 수정" → "버그를 재현하는 테스트를 쓰고, 통과시킨다"
- "X 리팩터링" → "리팩터링 전후로 테스트가 통과하는지 확인한다"

여러 단계 작업은 간단한 계획을 먼저 말한다:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

> PICKIT 적용 예: 방화벽(`checkSafetyGuardrail`)은 "동작한다"가 아니라 "±20% 초과·0원 이하·5개 이상·미지원 채널 케이스 각각에 대한 테스트가 통과한다"가 성공 기준이다. 예측 엔진(규칙 ⑥)은 "같은 입력 → 같은 숫자 출력(결정론적)"이 성공 기준이다.

## 9.5. No Closing Colons (한국어 출력)

**한국어 문장은 콜론이 아니라 마침표로 끝낸다.**

오너가 한국어로 쓰면 출력도 한국어다:
- 다음 줄이 리스트나 예시여도 문장을 `:`로 끝내지 않는다.
- 영어 문서로 학습한 LLM은 한국어에 콜론 습관이 샌다. 잡아낸다.
- 테스트: 모든 한국어 문장 종결부호는 `.`, `?`, `!` 중 하나여야 한다 — `:`가 아니다.
- 콜론은 코드·키값 쌍·라벨 안에서는 괜찮다. 문장 종결로만 쓰지 마라.

## 9.6. File Header Comments (한국어)

**모든 새 소스 파일의 첫 줄: 그 파일 역할을 한 줄 한국어 주석으로.**

새 파일을 만들 때:
- TypeScript/JavaScript: `// 쿠팡 주문 조회·수정을 PICKIT 단일 인터페이스로 변환하는 어댑터`
- Python: `# sales_log 일별 집계를 적재하는 배치 스크립트`
- SQL: `-- 상품별 판매량 시계열을 저장하는 테이블`
- 필수 지시어(`'use client'`, `'use server'`, shebang) 바로 아래에 둔다.
- 설정 파일(`*.config.ts`, `package.json` 등)은 건너뛴다.

이유: 에이전트는 코드베이스 전체가 아니라 파일을 선택적으로 읽는다. 한 줄 한국어 헤더가 즉각적인 맥락을 줘서, 다음 세션(사람이든 에이전트든)이 파일 전체를 다시 안 읽고도 길을 찾는다.

## 9.7. Plan + Checklist + Context Notes

**사소하지 않은 작업 전에 세 가지 산출물을 만든다. 이것 없이 코딩 시작 금지.**

- **Plan** — 무엇을 왜 만드는지.
- **Checklist** (`checklist.md`) — 구체적 작업을 체크박스로. 진행하며 체크.
- **Context Notes** (`context-notes.md`) — 작업 중 내린 결정과 그 이유. 계속 덧붙인다.

오너가 계획만 주고 바로 코딩하라고 하면, 멈추고 묻는다: "체크리스트와 컨텍스트 노트 먼저 만들까?" 다음 세션은 노트가 있어야 모든 결정을 다시 도출하지 않고 이어받는다.

## 9.8. Run Tests Before Marking Complete

**코드를 건드렸으면 "done" 하기 전에 테스트를 돌린다.**

- `npm test`, `pytest`, 프로젝트가 쓰는 게 뭐든 돌린다.
- 통과하면 결과를 보고한다. 실패하면 고치고 다시 돌린다.
- 테스트 셋업이 없으면 최소한 빌드·컴파일이 되는지 확인한다.
- 오너가 "끝", "완료", "다 됐어"라고 신호하기 전에 능동적으로 돌린다.

LLM이 가장 자주 건너뛰는 단계다. 타협 불가로 취급한다.

## 9.9. Semantic Commits

**하나의 논리적 변경이 끝나면 커밋한다. 오너가 시킬 때까지 기다리지 않는다.**

- 테스트: "이 커밋을 한 문장으로 설명할 수 있나?" 그렇다면 커밋. 아니면 변경이 아직 섞여 있다 — 쪼갠다.
- 좋음: "방화벽 금액 검증 추가". 나쁨: "방화벽 추가하고 ChatWidget UI도 고치고 버그도 수정" (3개로 쪼갠다).
- 관련 없는 편집 20개를 쌓아서 개별 롤백 능력을 잃지 않는다.
- 커밋을 위한 커밋은 하지 않는다 — 의미 있는 단위만.

참고: 솔로 프로토타입이나 일회성 스크립트는 느슨하게 묶어도 된다. 핵심은 의식이 아니라 되돌릴 수 있느냐다.

## 9.10. Read Errors, Don't Guess

**실제 에러·로그 줄을 읽는다. 기억으로 패턴 매칭하지 마라.**

뭔가 실패하면:
- 전체 에러 메시지와 스택 트레이스를 읽는다.
- 추측한 로그가 아니라 실제 로그 출력을 확인한다.
- 원인 확인 전에 '흔한 수정'을 적용하지 않는다.
- 불명확하면 print/log를 추가해 상태를 확인한 뒤 고친다.

LLM이 "테스트 실행" 다음으로 가장 자주 건너뛰는 단계다. 에러 키워드로 추측해 가장 최근 패턴 수정을 적용한다. 그게 한 줄 버그가 세 파일 리팩터링이 되는 경로다.
