# ONBOARDING_UI_SPEC.md — weave 온보딩 화면 명세

> 본 문서는 weave 온보딩(셀러 진입~채팅 시작) 구간의 UI 흐름·화면·디자인 규칙을 정의한다.
> `CLAUDE.md`(작업 규칙) 및 `PRD.md`(기능 명세)와 함께 참조한다. 기능 충돌 시 PRD.md가 기준이며, 온보딩 화면 구성·디자인은 본 문서가 기준이다.
> 본 문서는 기존 PRD `FR-001 온보딩`을 구체화·갱신한다.

---

## 0. 핵심 설계 원칙

- **답이 동작을 바꾸는 질문만 묻는다.** 사용자에게 묻는 모든 선택지는 제품 동작이나 기능 활성화를 실제로 분기시켜야 한다. 동작을 바꾸지 않는 질문은 단계만 늘리므로 제거한다.
- **마지막 장벽(API 연동)을 완화한다.** 채널 1개만 연동돼도 채팅을 시작할 수 있게 하고, 나머지는 진입 후 설정에서 보완하도록 유도한다.
- **첫인상을 성공/진행 프레이밍으로 구성한다.** 미설정 상태를 에러처럼 표시하지 않는다.

## 1. 전체 플로우

```
[진입]
  │
  ▼
1단계 · 재고 직접 보유 여부
  ├─ 예(직접 보유) ─────► 2단계 · 재고 관리 방식 ─► 3단계 · 채널 선택 ─► 4단계 · API 연결 ─► 연동 상태 ─► [채팅]
  └─ 아니오(위탁 판매) ─────────────(2단계 건너뜀)─────► 3단계 · 채널 선택 ─► 4단계 · API 연결 ─► 연동 상태 ─► [채팅]
```

- 직접 보유 셀러: 총 **4단계**
- 위탁 판매 셀러: 총 **3단계** (재고 관리 방식 화면 미노출)
- 기존 PRD에 있던 **"사입 셀러 / 자체 브랜드" 구분 질문은 온보딩에서 제거**한다. 두 유형 모두 "재고 직접 보유 = 예"에 해당하여 기능 분기가 발생하지 않으므로 묻지 않는다. (셀러 유형은 내부 타깃 정의용 분류로만 유지)

## 2. 공통 요소 (전 화면 적용)

### 2-1. 헤더 로고
- 화면 상단 중앙에 weave 원본 로고(SVG)를 배치한다.
- 로고는 **단일 컴포넌트(`<Logo />`)로 분리**하여 전 화면에서 재사용한다. 화면마다 SVG를 복사하지 않는다.
- 원본 SVG 사양: 아이콘은 단색 `#1A6FD4`(stroke-width 9, round cap)의 세 갈래 합류 형태이며, 워드마크 텍스트는 `#0A2540`이다. 임의로 화살촉을 추가하거나 색을 분리하지 않는다.
- 로고 텍스트 색(`#0A2540`)은 라이트 배경 기준 고정값이다.

### 2-2. 진행 표시 (통합 진행바)
- 단계 배지와 API 카운터를 분리하지 않고 **단일 진행바 하나로 통합**한다. (기존 "1/3단계"와 "API 연결 1/2"가 따로 노는 구조 폐기)
- 구성: 좌측 `N단계 · {단계명}`, 우측 `N / 총단계`, 하단 진행바.
- **분모(총 단계)는 1단계 응답에 따라 동적으로 결정**한다.
  - 직접 보유 선택 → 분모 4
  - 위탁 판매 선택 → 분모 3
- 4단계(API 연결)는 채널 수만큼 입력을 반복하므로, 메인 진행바 내부 진행률은 채널 입력 완료에 따라 점진 증가시킨다. (예: 마지막 단계 진입 시 부분 채움 → 채널 입력 완료 시 100%)

### 2-3. 디자인 토큰 (라이트모드 고정)
- **전 화면 라이트모드 고정.** `prefers-color-scheme: dark` 대응을 구현하지 않는다. 다크모드는 Post-MVP.

## 3. 화면별 명세

### 3-1. 1단계 · 재고 직접 보유 여부
- 질문: "재고를 직접 보유하고 계신가요?"
- 부제: "답에 따라 발주·재고 예측 기능을 켤지 정해요" (질문의 목적을 명시)
- 선택지 카드 2개(아이콘·제목·설명 모두 중앙 정렬):
  - **네, 직접 보유해요** → 발주·재고 예측(FR-006) 활성화, 2단계로 진행
  - **아니요, 위탁 판매예요** → 발주·재고 예측 비활성화, 2단계 건너뛰고 3단계로 진행
- 단일 선택이므로 카드 선택 즉시 다음 단계로 진행(별도 "다음" 버튼 없음).

### 3-2. 2단계 · 재고 관리 방식 (조건부: 직접 보유 시에만 노출)
- 질문: "재고를 어떻게 관리하시나요?"
- 부제: "관리 방식에 따라 동기화 기능이 달라져요"
- 선택지 카드 2개(아이콘 중앙 정렬, 1단계와 동일 스타일):
  - **공유 창고 운영**: 모든 채널이 같은 재고 공유. 한 채널 판매 시 타 채널 재고 자동 차감(단방향 푸시 동기화 = 기본 활성).
  - **채널별 분리 관리**: 채널마다 재고를 분리 배정. 자동 동기화 비활성, 채널별 발주 예측 제공.
- 화면 하단 고정 안내: "관리 방식은 나중에 설정에서 언제든 바꿀 수 있어요" (아이콘: settings)
- **중요 — 변경 동선**: 재고 관리 방식 전환은 채팅 명령으로 즉시 토글하지 않는다. 채팅 화면의 **설정 탭**에서만 변경한다.
  - 사유: 공유↔분리 전환은 단순 ON/OFF가 아니라 재고 데이터 재배치(분리 시 채널별 배분량 산정, 공유 시 합산)가 필요하다. 채팅 명령 1회로 처리 시 초과판매·품절 사고 위험이 있어 방화벽 철학과 충돌한다.
  - 채팅에서 관련 요청 입력 시 → 설정 탭으로 안내하는 처리만 수행한다.

### 3-3. 3단계 · 채널 선택
- 제목: "운영 중인 채널을 선택해 주세요"
- 부제: "2개 이상 · 나중에 설정에서 언제든 변경할 수 있어요"
- 채널 6개(쿠팡 / 네이버 스마트스토어 / 카페24 / 무신사 / 에이블리 / 지그재그)를 3×2 그리드 카드로 배치.
- **채널명 앞 색상 점(dot) 제거.** 채널 구분은 명칭으로 충분하며, 상태는 선택 테두리·배지로 표현한다.
- **"조회·송장" 인라인 배지 제거.** 대신 의류 버티컬 3사(무신사·에이블리·지그재그) 선택 시 **안내 모달**을 노출한다.
- 복수 선택(토글)이므로 2개 이상 선택 시 활성화되는 "다음 — API 연결하기" 버튼을 별도 제공한다.

#### 3-3-1. 의류 3사 안내 모달
- 트리거: 무신사·에이블리·지그재그 카드 선택 시.
- 구성: 정보 아이콘 + "{채널명}은 일부 기능만 지원해요" + 지원/미지원 구분.
  - 지원(체크): 주문 조회 · 송장 전송
  - 미지원(X, 회색): 가격·재고 수정, 취소·반품 → "파트너센터에서 직접 처리"
- 버튼: "취소" / "알겠어요, 선택"
- **채널별 추가 문구**:
  - 에이블리: "셀러스(Sellers) 입점 타입만 연동 가능합니다. 입점 타입을 먼저 확인해 주세요." (PRD 필수 안내 사항)
- (선택적 UX) 동일 채널에 대해 모달은 최초 1회만 노출하고, 이후에는 카드 내 작은 배지로만 표시하는 방식 검토 가능.

### 3-4. 4단계 · API 연결
- 단계 라벨: "4단계 · API 연결 (마지막)"
- 선택한 채널 수만큼 입력을 반복한다. 상단에 **채널 서브 스텝**(예: `쿠팡 → 네이버 / 1·2 채널`)을 표시하여 메인 진행바와 별개로 채널 진행을 안내한다.
- 각 채널 입력부 구성:
  - 발급 위치 가이드 + 외부 발급 페이지 바로가기 링크(예: 쿠팡 Wing)
  - 채널별 인증 입력 필드 (아래 3-4-1 참조)
  - 보안 안내(고정): "입력한 키는 암호화되어 안전하게 보관되며, 화면에 다시 표시되지 않아요" (lock 아이콘)
  - 버튼: "이 채널 건너뛰기"(보조) / "저장하고 다음 →"(주)
  - 하단 안내: "건너뛰어도 채팅 화면 설정에서 언제든 입력할 수 있어요"
- 채널 1개라도 연동되면 온보딩 완료 및 채팅 진입이 가능하다.

#### 3-4-1. 채널별 인증 폼 차이 (구현 시 분기 필수)
- 채널마다 필요한 인증 정보가 달라 폼 구조가 달라진다. 단일 폼으로 통일하지 않는다.

| 채널 | 인증 방식 | 입력 형태 |
| --- | --- | --- |
| 쿠팡 | Vendor ID + Access Key + Secret Key | 텍스트 입력 3종 (Secret Key는 password 타입) |
| 네이버 스마트스토어 | 커머스API Application ID + Secret | 텍스트 입력 2종 |
| 카페24 | OAuth 인증 | 입력 필드가 아닌 "연결하기" 버튼 → 로그인 리다이렉트 (UX 상이) |
| 무신사 / 에이블리 / 지그재그 | 각 파트너 API 방식 | **MVP 착수 전 각 플랫폼 API 문서·제휴 확인 후 확정** |

- 무신사: API Key 발급 후 1년 만료 정책 → 만료 갱신 알림 로직 고려(FR-008 연계).

### 3-5. 완료 · 연동 상태 화면
- 헤드라인: 부정 카운트("0개 정상") 금지. **진행/성공 프레이밍** 사용 (예: "연결 준비 완료", "N개 채널 중 M개 연결됨 · 지금 바로 시작할 수 있어요").
- 채널 카드 상태를 **3가지로 명확히 구분**한다:
  - **연결됨**: 성공 톤(연한 초록 배경 + 초록 테두리 + 중앙 체크 + "연결 완료"). 기능 나열 텍스트는 표시하지 않는다.
  - **연결 대기**: 중립 톤(회색 배지 "연결 대기"). 미설정은 에러가 아니므로 빨간색·"만료" 표기 금지. "연결하기" 버튼 제공.
  - **연결 만료/실패**: 빨간 톤. 실제 토큰 만료·인증 실패 시에만 사용. "다시 연결하기" 제공.
- 주의: "만료" 표기는 실제 토큰 만료 시에만 사용한다. 최초 온보딩(미입력) 상태에는 사용 금지.
- CTA: "채팅 시작하기"(채널 1개 이상 연결 시 활성) + 하단 안내 "나머지 채널은 채팅 화면에서 언제든 연결할 수 있어요".

## 4. 구현 체크리스트

- [ ] 사입/자사 구분 질문 제거 (1단계는 재고 보유 여부만)
- [ ] 진행바 단일화 + 분모 동적 처리(직접 보유 4 / 위탁 3)
- [ ] 위탁 판매 선택 시 2단계(재고 관리 방식) 스킵
- [ ] 재고 관리 방식 변경은 채팅 설정 탭 동선으로만 연결 (채팅 명령 즉시 토글 금지)
- [ ] 채널 선택 화면: 색점 제거, 인라인 "조회·송장" 배지 제거
- [ ] 의류 3사 선택 시 안내 모달(에이블리는 셀러스 타입 안내 추가)
- [ ] API 단계: 채널 서브 스텝 + 채널별 폼 분기(쿠팡 3키 / 네이버 2키 / 카페24 OAuth)
- [ ] API 보안 안내 문구 + 채널 단위 "건너뛰기"
- [ ] 연동 상태: 성공/대기/에러 3상태 분리, "만료" 오표기 금지, 성공 프레이밍
- [ ] 디자인 토큰 전역 정의, 라이트모드 고정(다크 대응 미구현)
- [ ] 로고 단일 컴포넌트화(원본 SVG)
