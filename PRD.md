## 1. 제품 개요 (Product Overview)

### 1-1. 한 줄 소개

PICKIT은 "엑셀 관리는 한계에 왔지만, 비싼 ERP를 쓰기엔 아직 이른" 의류 멀티채널 셀러를 위한 웹/모바일 자연어 기반 멀티채널 운영 서비스입니다.

### 1-2. 배경 및 문제 정의

- 기존 ERP(사방넷, 이지어드민 등)는 창고 관리·바코드 시스템을 갖춘 대형 법인 셀러 중심이라 소규모 셀러에겐 가입비·교육비 부담이 큼.
- 의류 셀러는 색상×사이즈 옵션이 많고 시즌 회전이 빠르며, 복수 채널에 흩어진 주문·재고를 엑셀로 수동 관리 → 발주 누락, 초과판매, 반복 노동이 일상적으로 발생.
- 멀티채널을 자연어로 제어하는 도구가 부재하여 이 빈틈을 공략.

### 1-3. 비즈니스 목표

- 의류 멀티채널 셀러 대상 PMF(제품-시장 적합성) 확보
- 무료 체험 → 유료 전환율 확보 (목표치 TBD)
- 초기 채널 커버리지: 6개 (쿠팡·네이버·카페24·무신사·에이블리·지그재그)

## 2. 타깃 사용자 (Target Users)

| **B. 자체 브랜드** | 자체 제작 상품, 직접 재고 보유 | 낮은 이탈률, 장기적 안정 매출 |

## 4. 기능 요구사항 (Functional Requirements)

### FR-001: 온보딩 — 셀러 유형 및 채널 선택

**우선순위**: P0

**목적**: 진입 시 셀러 유형을 파악하여 기능 활성화 범위를 결정하고, 연동 채널 및 주의사항을 안내한다.

**상세 동작**:

1. 최초 진입 시 단 하나의 질문: "재고를 직접 보유하고 계신가요?"
    - Yes → 후속 질문(2번)으로 이동
    - No → 발주 예측 비활성화 + "추후 재고 도미노 기능 지원 예정" 안내 후 채널 선택으로 이동
2. **[hasInventory=Yes일 때만]** 후속 질문: "재고를 한 창고에서 모든 채널이 공유하나요, 아니면 채널별로 나눠서 관리하나요?"
    - 공유(shared) → 재고 자동 동기화(FR-009) 활성화. 발주 예측은 전체 기준 단일 예측.
    - 분리(split) → 재고 자동 동기화 비활성화. 발주 예측은 채널별 독립 예측. 조회·송장·가격수정 등 나머지 기능은 정상 작동.
    - 결과를 `stockMode('shared' | 'split')`으로 저장
3. 연동 채널 복수 선택: 쿠팡 / 네이버 스마트스토어 / 카페24 / 무신사 / 에이블리 / 지그재그
    - 에이블리 선택 시: "셀러스(Sellers) 입점 타입만 연동 가능합니다. 입점 타입을 먼저 확인해 주세요." 안내 팝업 노출
4. 유형·채널은 설정 메뉴에서 언제든 변경 가능

> **MVP 범위**: stockMode는 전역 설정 하나. 상품 단위로 공유/분리를 섞는 예외 처리는 Phase 2.

**Acceptance Criteria**:

- [ ] 재고 보유 Yes 선택 시, 공유/분리 재고 후속 질문이 표시된다.
- [ ] 공유 재고 선택 시, 재고 자동 동기화(FR-009)가 활성화된다.
- [ ] 분리 재고 선택 시, 재고 자동 동기화가 비활성화되고 나머지 기능(조회·송장·가격수정)은 정상 작동한다.
- [ ] 재고 보유 No 선택 시, 발주 예측 알림이 비활성화되고 추후 기능 안내가 노출된다.
- [ ] 에이블리 선택 시, 셀러스 타입 확인 안내가 반드시 표시된다.

### FR-002: 통합 주문 조회 (READ)

**우선순위**: P0

**목적**: 6개 채널에 흩어진 주문 현황을 자연어 한 문장으로 통합 조회한다.

**상세 동작**:

1. 자연어 명령으로 채널·기간 지정하여 조회
2. 기간 키워드 미지정 시: 자동으로 당일(00:00~현재) 기준 적용 (서버 과부하 방지)
3. 의류 버티컬 3사의 취소·반품·교환 처리 기능은 미지원 → 조회 결과와 함께 각 플랫폼 파트너센터 바로가기 링크 제공

**조회 제공 항목**:

| 항목 | 포함 데이터 |
| --- | --- |
| 들어온 주문 | 상품명, 옵션(사이즈·색상), 수량 |
| 취소된 주문 | 상품명, 옵션, 수량, 취소 사유 |
| 반품된 주문 | 상품명, 옵션, 수량, 반품 상태 |
| 교환된 주문 | 상품명, 옵션, 수량, 교환 상태 |

**채널별 처리 지원**:

- 쿠팡, 네이버 스마트스토어, 카페24: 조회 및 처리(취소·반품·교환 승인) 모두 지원
- 무신사, 에이블리, 지그재그: **조회만 지원**. 취소·반품·교환 처리는 각 플랫폼 파트너센터 링크로 안내

**Acceptance Criteria**:

- [ ] 기간 키워드 없이 "주문 보여줘" 입력 시, 당일 00:00~현재 기준 주문이 반환된다.
- [ ] 의류 버티컬 채널의 반품 건 조회 시, 조회 결과 + 파트너센터 링크가 함께 제공된다.
- [ ] 채널 필터 미지정 시, 연동된 전체 채널 주문이 통합 조회된다.

### FR-003: 일괄 품절·가격 수정 (WRITE)

**우선순위**: P0

**목적**: 여러 채널의 상품 가격·재고·품절 상태를 자연어 명령 한 번으로 수정한다.

**채널별 지원 범위**:

**[쿠팡, 네이버 스마트스토어, 카페24]** — 실시간 수정 지원

- 가격 변경, 옵션별 재고 수정, 즉시 품절 처리
- 처리 완료 후 해당 채널 수정 페이지 바로가기 링크 제공

**[무신사, 에이블리, 지그재그]** — MVP에서 직접 수정 보류

- 플랫폼별 외부 쓰기(Write) API 범위 확인 중
- MVP에서는 해당 채널 상품 수정 페이지 바로가기 링크 제공으로 우회
- API 확인 완료 후 기능 순차 활성화 예정

**Acceptance Criteria**:

- [ ] "쿠팡이랑 네이버에서 ○○ L 품절시켜줘" 입력 시, 방화벽 통과 후 두 채널 모두 품절 처리되고 각 채널 수정 페이지 링크가 제공된다.
- [ ] 무신사 수정 명령 시, "현재 무신사는 직접 수정을 지원하지 않습니다. 아래 링크에서 직접 처리해 주세요." + 무신사 파트너센터 링크가 제공된다.
- [ ] 방화벽 조건 해당 시, 실행이 보류되고 승인 요청 UI가 렌더링된다.

### FR-004: 일괄 송장 전송

**우선순위**: P0

**목적**: 셀러의 핵심 반복 노동인 "주문 수집 후 송장 번호를 각 쇼핑몰에 재입력"을 말 한마디로 처리한다.

**상세 동작**:

1. 수집된 주문 목록에서 택배사 연동 또는 송장 번호 직접 입력
2. 자연어 명령으로 일괄 전송 또는 채널·주문 지정 전송
3. 6개 채널 전체 송장 전송 지원 (각 플랫폼 API 확인 범위 내)

**Acceptance Criteria**:

- [ ] 송장 번호가 입력된 주문 목록에서 "오늘 주문 전체 송장 전송해줘" 명령 시, 6개 채널에 일괄 전송된다.
- [ ] 특정 채널만 지정 전송이 가능하다.
- [ ] 전송 완료 후 채널별 처리 결과가 표시된다.

### FR-005: 주문 데이터 엑셀 추출

**우선순위**: P1

**목적**: 채널별 주문·매출 데이터를 자연어 명령으로 즉시 엑셀 파일로 다운로드한다.

**상세 동작**:

1. 자연어로 채널·기간 지정
2. 백엔드에서 정제된 JSON 데이터 반환
3. 프론트엔드에서 클라이언트 사이드(`xlsx` 라이브러리)로 파일 생성 → 다운로드 링크 제공 (서버 비용 최소화)

**응답 예시**: "6월 1일 기준 이번 달 네이버 주문·정산 데이터 총 142건을 정리했습니다. 💾 [2026_06_네이버_매출_리포트.xlsx 다운로드]"

**Acceptance Criteria**:

- [ ] 채널·기간 지정 시, 해당 조건의 주문 데이터가 .xlsx 파일로 생성된다.
- [ ] 기간 미지정 시, 당일 기준으로 처리된다.
- [ ] 파일 생성은 프론트엔드(클라이언트)에서 이루어지고 서버에 파일이 저장되지 않는다.

### FR-006: 재고 소진 알림 (발주 예측 MVP)

**우선순위**: P1

**활성화 조건**: FR-001 온보딩에서 "재고 직접 보유 = Yes" 선택한 셀러만

**목적**: 누적 판매 데이터를 기반으로 최대 4개월 뒤 재고 소진 시점을 예측하여 적시 발주를 유도한다.

**예측 로직**:

| 단계 | 계산 방식 |
| --- | --- |
| 기본 판매 속도 | 최근 4개월 총 판매량 ÷ 날짜 수 |
| 재구매 가중치 | 상품별 시계열 파동 분석 (회원 식별 불필요, 전체 주문 집계 기반으로 소모 주기 통계 산출) |
| 4개월 예측 수요 | (기본 판매 속도 × 120일) + 집계 파동 기반 재구매성 예측 물량 |
| 시즌 가중치 | 의류 특성 반영 (계절성 성수기·비수기 가산) |

> 쿠팡·무신사 등 회원 정보를 알 수 없는 채널, 자사몰의 비회원 주문도 전체 판매량 집계로 대응 가능.

**stockMode별 예측 경로**:

| stockMode | currentStock 소스 | 예측 방식 |
| --- | --- | --- |
| shared (공유 재고) | 임의 한 채널의 `getStock` 값 (전 채널 동일) | 전체 채널 판매속도 합산 → 단일 예측 |
| split (분리 재고) | 채널별 `getStock` 값 | 채널별 독립 예측 — 4개월 이내 소진 채널만 알림 |

> 분리 재고 셀러는 채널마다 소진 시점이 다를 수 있어 채널별로 개별 예측한다. 카드 과다 방지를 위해 4개월(120일) 이내 소진이 예측되는 채널만 알림에 표시한다.

**알림 예시**: "○○ 상품 L, 현재 판매 추세 기준 10월 초 재고 완전 소진 예측. 안정적인 물량 확보를 위해 지금 발주를 준비하세요. 권장 수량 ○○개"

**데이터 전제**: `sales_log` 테이블에 데이터가 누적되어야 정확도 확보. **MVP 배포 즉시 적재 시작 필수** (적재 지연 시 예측 기능 영구 불가).

**Acceptance Criteria**:

- [ ] 재고 보유 Yes 셀러에게만 발주 알림이 활성화된다.
- [ ] 공유 재고(shared) 셀러는 전체 기준 단일 소진 예측과 권장 수량이 제공된다.
- [ ] 분리 재고(split) 셀러는 채널별 소진 예측이 제공되고, 4개월 이내 소진이 임박한 채널만 알림 카드로 표시된다.
- [ ] `sales_log`에 데이터가 없는 신규 사용자에게는 "데이터 누적 중" 안내가 표시된다.

---

### FR-007: 방화벽 (4종 안전장치)

**우선순위**: P0

**목적**: 자연어 처리의 오작동으로 인한 대규모 수정 사고를 방지한다. AI가 산출한 수정 값을 실제 API 전송 직전, 백엔드 인터셉터에서 `checkSafetyGuardrail()` 함수로 검사한다.

**방화벽 4종**:

| 종류 | 트리거 조건 | 처리 방식 |
| --- | --- | --- |
| **금액 방화벽** | 기존 단가 대비 ±20% 초과 또는 최종 단가 0원 이하 (할인·쿠폰 포함) | 보류 + 승인 요청 UI |
| **범위 방화벽** | 대상 상품 5개 이상 또는 '전체·모두·싹 다·일괄' 키워드 포함 | 보류 + 승인 요청 UI |
| **맥락 이해 방화벽** | 의도 파싱 Confidence Score < 85% 또는 정보 부족 | 되물음 처리 |
| **플랫폼 불일치 방화벽** | 해당 채널이 지원하지 않는 기능 요청 | 가능 채널 처리 + 불가 채널 링크 안내 |

**UI 동작 플로우**:

1. 방화벽 트리거 → `isDangerous: true` 반환
2. 실제 반영 보류(Hold)
3. 채팅창 내 독립 경고 블록 + `[최종 승인하기] / [취소]` 버튼 렌더링
4. 셀러 승인 전까지 백엔드 API 요청 미실행

> **참고**: Amazon은 2026년 3월 신규 규정으로 AI의 24시간 내 20% 초과 자동 가격 변경에 인간 승인을 의무화했습니다. PICKIT의 금액 방화벽이 글로벌 규제 방향과 정확히 일치합니다.

**Acceptance Criteria**:

- [ ] 기존가 10,000원인 상품을 8,000원 이하(20% 초과 인하)로 변경 시, 자동 실행되지 않고 승인 요청 UI가 표시된다.
- [ ] "모든 상품 가격 500원 내려줘" 명령 시, 범위 방화벽이 트리거되어 보류된다.
- [ ] 승인 버튼 클릭 전까지 API 요청이 발생하지 않는다.
- [ ] 지그재그에서 지원하지 않는 기능 요청 시, 해당 채널 링크와 함께 불가 사유가 안내된다.

---

### FR-008: 연동 끊김 안내

**우선순위**: P1

**목적**: 외부 플랫폼의 비밀번호 변경, API 토큰 만료, 일시적 서버 다운 발생 시 서비스 먹통으로 오해하지 않도록 원인과 해결법을 즉시 안내한다.

**상세 동작**:

- 연동 에러 코드 감지 → 해당 채널 작업 홀딩 → 주황색 경고 말풍선 표시
- 경고 말풍선 내 [연동 설정 페이지] 바로가기 하이퍼링크 제공
- 안내 예시: "현재 [네이버 스마트스토어] 연결이 만료되었습니다. 우측 상단 '연동 설정 페이지'에서 다시 연결해 주세요."

> **무신사 특이사항**: 2026년 3월 API 정책 변경으로 API Key 유효 기간이 발급일로부터 1년(365일)으로 변경. 만료 전 갱신 알림 자동화 권장.

**Acceptance Criteria**:

- [ ] API 토큰 만료 시, 해당 채널에서만 에러가 격리되고 나머지 채널은 정상 운영된다.
- [ ] 에러 발생 채널은 주황색 경고와 함께 재연동 안내가 노출된다.

---

### FR-009: 재고 자동 동기화

**우선순위**: P1

**목적**: 한 채널에서 판매 발생 시 다른 채널의 재고를 자동으로 차감하여 초과판매(Oversell)를 방지한다.

**활성화 조건**: FR-001 온보딩에서 `stockMode = 'shared'`(공유 재고)를 선택한 셀러만. 분리 재고(`split`) 셀러는 채널별 배분이 다르므로 자동 push를 하지 않는다. 조회·송장·가격수정 등 나머지 기능은 분리 재고 셀러도 정상 작동한다.

**MVP 지원 범위**:

| 채널 | 상태 |
| --- | --- |
| 쿠팡, 네이버 스마트스토어, 카페24 | 공유 재고 모드에서 단방향 푸시 동기화 구현 |
| 무신사, 에이블리, 지그재그 | 쓰기(Write) API 범위 확인 후 단계적 활성화 |

**구현 방식**: 판매 감지(query 결과) → 소스 채널 재고 API 실시간 조회(`getStock`) → 나머지 핵심 채널에 현재 재고값 push. 멱등성 보장(같은 쿼리를 두 번 해도 현재 재고를 두 번 push할 뿐, 이중 차감 없음).

> **셀러 안내 필수**: 공유 재고 모드에서 각 쇼핑몰 어드민에서 직접 재고를 수정할 경우 PICKIT과 불일치가 발생할 수 있음. PICKIT을 통한 단일 경로 관리 권장.
> **상품 단위 공유/분리 예외**: Phase 2. MVP는 전역 stockMode 하나로 운영.

**Acceptance Criteria**:

- [ ] 공유 재고 모드에서 쿠팡에서 L 사이즈 판매 시, 쿠팡 현재 재고를 조회하여 네이버·카페24에 자동 동기화된다.
- [ ] 분리 재고 모드에서는 자동 동기화가 실행되지 않고, 조회·송장·가격수정 기능은 정상 작동한다.
- [ ] 재고 동기화 실행 내역이 로그로 기록된다.

## 5. 비기능 요구사항 (Non-Functional Requirements)

### 5-1. 성능

- 자연어 명령 → 응답: 일반 조회 3초 이내 목표
- 외부 API 호출 실패 시: 최대 3회 재시도 후 에러 처리
- 대량 호출(Bulk Request) 방화벽 트리거 전 사전 차단으로 서버 과부하 방지

### 5-2. 보안

- 각 플랫폼 API Key·Token은 서버 측 암호화 저장
- 사용자 주문 데이터는 서버에 장기 캐시하지 않고 실시간 조회 원칙
- 방화벽 파이프라인: API 전송 직전 `checkSafetyGuardrail()` 반드시 통과

### 5-3. 가용성

- 특정 채널 API 오류 발생 시, 해당 채널만 격리하고 나머지 채널은 정상 운영
- 각 채널 장애를 서비스 전체 장애로 오인하지 않도록 상태 분리

### 5-4. 확장성

- 신규 채널 추가 시 `src/connectors/`에 파일 추가만으로 확장 가능 (어댑터 패턴 유지)
- AI 플러그인·코어 엔진 변경 없이 채널 독립 확장

## 6. 기술 아키텍처 (Technical Architecture)

### 규칙 ①: AI와 쇼핑몰 API의 완전 격리 (어댑터 패턴 필수)

- AI(LLM)에게는 단 하나의 통합 마스터 함수 `modify_product_data(platform, productId, action, value)` 만 Tool로 제공
- 백엔드가 `platform` 인자를 분기 처리하여 실제 커넥터 파일(`src/connectors/coupang.js`, `src/connectors/naver.js` 등)을 트리거
- 채널 추가 시 새 커넥터 파일만 추가하면 AI·코어 변경 불필요 (플러그인 구조)
- **토큰 효율**: AI에게 채널별 API 명세를 모두 노출하지 않아 LLM 컨텍스트 최소화

### 규칙 ②: 3중 유효성 검증 가드레일 (방화벽 파이프라인)

- AI 수정 값을 실제 API로 전송하기 직전, 백엔드 인터셉터에서 `checkSafetyGuardrail()` 반드시 실행
- 트리거 시: `isDangerous: true` 반환 → Hold → 승인 요청 UI
- 3가지 검사: ① 금액(±20%, 0원 이하) ② 범위(5개 이상, 일괄 키워드) ③ 확신도(Confidence Score < 85%)

### 규칙 ③: 시스템 연동 예외 처리 및 기본값 필터

- 연동 에러 코드 수신 시: 해당 채널 작업 홀딩 + 주황색 경고 + [연동 설정 페이지] 바로가기 제공
- 기간 키워드 미지정 조회: 자동으로 `[당일: 오늘 00:00~현재]` 범위만 호출 (서버 과부하 방지)

### 규칙 ④: 프론트엔드 스타일 스코프 격리 및 연산 분산

- `ChatWidget.jsx`: CSS Modules 또는 Styled-components로 스타일 철저히 격리 + `position: fixed; z-index: 9999;`
- 엑셀 생성: 백엔드는 청정 JSON만 반환, 프론트엔드에서 `xlsx` 라이브러리로 셀러 PC 자원 활용 파일 생성 → 서버 비용 최소화

### 규칙 ⑤: 예측·CRM 데이터 스키마 분리

| 테이블 | 용도 | 적재 시점 |
| --- | --- | --- |
| `sales_log` | 상품별 집계 판매량 시계열 (발주 예측용, 개인정보 불필요) | **MVP 배포 즉시 시작** |
| `customer_purchase` | 회원 개인별 구매 이력 (추후 CRM용) | Post-MVP, 회원 데이터 확보 시 |

> **원칙**: 예측은 기능이 아니라 쌓인 데이터의 결과물. `sales_log` 적재 시작이 늦을수록 예측 정확도 확보도 늦어짐.

## 11. 추후 개발 로드맵 (Post-MVP)

### Phase 2

- 조건부 예약 자동화: 시간 조건·재고 도미노·경쟁가 추적
- 의류 3사 실시간 수정 활성화 (API 확인 완료 후 순차 적용)
- 발주 예측 고도화: 이동평균 + 시즌 가중치 정교화
- 운영 대시보드: 옵션(색상×사이즈)별 판매 랭킹·죽은 재고 식별 + 매일 아침 챗 요약(굿모닝 리포트). 기존 주문 조회(FR-002) 기능을 시각화로 재활용하는 가벼운 범위
    - *벤치마크 참고*: 씨그로(cigro) 등 이커머스 대시보드는 "보는 도구(분석)" 중심이나, PICKIT은 "하는 도구(운영 액션)" 정체성을 유지하며 옵션 단위 시각화로 차별화

### Phase 3

- 시계열 예측 모델 고도화 (시즌·이벤트·재구매 AI 학습)
- 위탁·드롭쉬핑 전용 재고 도미노 트리거
- 재구매 마케팅 CRM (회원결제 비중 충분한 B 유형 브랜드)
- 발주 예측 시각화 + 손익 지표: `sales_log`가 충분히 적재된 뒤 활성화. 광고비는 미연동이므로 수수료·배송비 기반의 정직한 손익 지표만 제공(완전 공헌이익은 BM 범위 밖)

### Phase 4 (글로벌)

- 동남아 채널 우선 진출: 쇼피(Shopee), 라자다(Lazada)
- 국내에서 검증한 방화벽 UX·발주 예측 기능을 동남아 의류 셀러 시장에 이식

> **미국 시장**: Linnworks·PROPAMP 등 기존 도구 포화로 단순 통합·예측만으로는 차별화 어려움. 동남아 이후 재검토.
